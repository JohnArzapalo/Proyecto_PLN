Vista General

El RAG (Retrieval-Augmented Generation) es el componente de conocimiento del sistema Hannah. Su función es recuperar información relevante de una base de datos vectorial y formatearla como contexto para que los modelos generativos (Fast y Slow) produzcan respuestas informadas.

Cambio importante (abril 2026)

El Slow Model cambió de Meta Llama a Qwen2.5-14B-Instruct con Supervised Fine-Tuning (SFT) aplicado por el equipo. Esto se refleja en el Laboratorio Grupal.pdf (9 de abril 2026). El RAG no se ve afectado por este cambio ya que es agnóstico al modelo — solo entrega contexto formateado.

El RAG es standalone: no requiere que los modelos (Hannah 360M o Qwen2.5-14B) estén corriendo. Recibe una query de texto y una señal de modo (simplified o extended) y devuelve contexto formateado con tokens [MEMORY]...[/MEMORY].

Primer Entregable (24 abril 2026)

Según Marlow: "como primer entregable, un transformador y BD para el Fast (Hannah menor de edad) funciona." Es decir, el modo Simplified debe estar 100% funcional. El modo Extended puede estar estructuralmente preparado pero no es obligatorio que esté perfecto.

Componentes del Pipeline

1. EmbeddingService Ambos modos

Convierte texto en vectores numéricos de 384 dimensiones usando all-MiniLM-L6-v2 (~80MB, CPU).

2. VectorStore Ambos modos

Base de datos vectorial con ChromaDB. Persistencia local, búsqueda por similitud coseno (HNSW).

3. SemanticCache Ambos modos

"Memoria torpe" (como lo describió Marlow). Si una query tiene similitud ≥0.92 con una anterior, devuelve el resultado cacheado sin hacer búsqueda vectorial.

4. QueryEnhancer Solo Extended

Mejora las queries antes de buscar. En Simplified solo limpia; en Extended genera expansiones y un documento HyDE.

5. ContextHandler Ambos modos

Procesa los chunks recuperados: trunca al límite, opcionalmente rerankea, y formatea con [MEMORY]/[/MEMORY].

6. RAGComponent Orquestador

Punto de entrada principal. Une todos los componentes y expone retrieve(query, mode) y aretrieve(query, mode) (async).

embeddings.py

Estado: Implementado ✅

Modelo: all-MiniLM-L6-v2 (Sentence Transformers). 384 dimensiones, ~80MB, corre en CPU.

from sentence_transformers import SentenceTransformer

class EmbeddingService:
    def __init__(self, model_name="all-MiniLM-L6-v2"):
        self.model = SentenceTransformer(model_name, device="cpu")

    def get_embedding(self, text: str) -> list[float]:
        # normalize_embeddings=True → dot product = cosine similarity
        return self.model.encode(text, normalize_embeddings=True).tolist()

    def get_embeddings_batch(self, texts: list[str]) -> list[list[float]]:
        return self.model.encode(
            texts, normalize_embeddings=True,
            show_progress_bar=True, batch_size=32
        ).tolist()

Decisiones clave:

Se usa CPU explícitamente (device="cpu") para no competir con VRAM cuando se integre con Hannah. La normalización está activada porque ChromaDB usa distancia coseno, y con vectores normalizados, el dot product equivale a similitud coseno (más eficiente). El batch_size de 32 balancea velocidad vs. memoria RAM.

vector_store.py

Estado: Implementado ✅

import chromadb
from embeddings import EmbeddingService

class VectorStore:
    def __init__(self, db_path="./hannah_vectordb"):
        self.client = chromadb.PersistentClient(path=db_path)
        self.embedder = EmbeddingService()
        self.collection = self.client.get_or_create_collection(
            name="knowledge_base",
            metadata={"hnsw:space": "cosine"}  # IMPORTANTE: default es L2
        )

Decisión crítica: ChromaDB usa distancia L2 por defecto, NO coseno. Sin el parámetro {"hnsw:space": "cosine"}, los resultados serían incorrectos porque nuestros embeddings están normalizados para similitud coseno.

¿Por qué 0.92? Es un umbral que balancea entre no retornar falsos positivos (preguntas distintas que suenan parecido) y capturar paráfrasis reales. En pruebas con sentence-transformers, queries que son paráfrasis genuinas suelen tener similitud >0.90, mientras que queries semánticamente distintas quedan por debajo de 0.85.

query_enhancer.py

Estado: Nuevo 🆕

Implementa dos técnicas de mejora de queries:

Técnica Descripción Cuándo se usa
Query Expansion Genera variantes: declarativa, keywords, contextual Solo Extended
HyDE (templates) Documento hipotético basado en tipo de pregunta Solo Extended

HyDE sin LLM: La implementación original de HyDE (Gao et al., 2022) usa un LLM para generar el documento hipotético. Como nuestro RAG es standalone, usamos templates categorizados por tipo de pregunta (definición, how-to, comparación, factual). Esto captura ~70% del beneficio de HyDE sin necesidad de un modelo generativo.

Teoría: RAG

Retrieval-Augmented Generation (Lewis et al., 2020) combina un retriever (que busca documentos relevantes) con un generator (que produce texto). En vez de depender solo de lo que el modelo "aprendió" durante el entrenamiento, el RAG le inyecta conocimiento externo en tiempo de inferencia.

En nuestro caso, el retriever es ChromaDB + all-MiniLM-L6-v2, y los generators son Hannah 360M (Fast) y Qwen2.5-14B-Instruct (Slow). El RAG no genera texto — solo recupera y formatea contexto. La generación la hacen los modelos.

Teoría: HyDE

Hypothetical Document Embeddings (Gao et al., 2022) resuelve el problema de query-document mismatch: una pregunta como "¿Qué es NLP?" tiene un embedding muy diferente al de un documento que explica NLP, porque la pregunta es corta y el documento es largo y detallado. HyDE genera un "documento hipotético" que respondería la pregunta, y usa ESE embedding para buscar. El embedding del documento hipotético está más cerca de los documentos reales en el espacio vectorial.

Nuestra implementación usa templates en vez de un LLM para generar el documento hipotético, lo que lo hace lightweight y no requiere modelo adicional.

Teoría: Query Expansion

La expansión de queries genera múltiples variantes de la pregunta original para cubrir más espacio semántico en la búsqueda. Nuestras estrategias:

Estrategia Ejemplo Por qué funciona
Pregunta → Afirmación "¿Qué es NLP?" → "NLP" Documentos suelen ser afirmativos
Extracción de keywords "¿Cómo entreno DPO?" → "entrenar DPO" Elimina ruido de stopwords
Contextualización "¿Qué es NLP?" → "Hannah chatbot ¿Qué es NLP?" Sesga hacia documentos del dominio

Teoría: Prepared Mind, Fast Response

PMFR (Zhang et al., 2025) propone desacoplamiento temporal: mientras un componente decide (Model Selector), otro ya puede estar trabajando en paralelo (RAG). Nuestra implementación async (aretrieve()) permite exactamente esto: el RAG puede comenzar la búsqueda antes de que el Model Selector termine de decidir, y luego adaptar el modo cuando llega la señal.

Correcciones vs Análisis Gemini

Durante el desarrollo, se consultó con Gemini (Google) sobre la arquitectura. A continuación se documentan los puntos donde Gemini acertó y donde se equivocó, para que el equipo tenga claridad sobre las decisiones tomadas.

Lo que Gemini acertó

1. CPU para embeddings: Correctamente identificó que el RAG debe correr en CPU para no competir con la VRAM de la RTX 5070 Ti.
2. System Prompt de AI Companion: Identificó que Hannah es una "Novia Virtual", no un asistente genérico. El DPO castiga respuestas tipo "Soy un asistente de IA".
3. Pipeline async: Correcto que el orquestador debe ser asíncrono con ThreadPoolExecutor.
4. Tokens especiales: Correctamente identificó [SYS], [USR], [ASS] con add_special_tokens=False.

Donde Gemini se equivocó

1. SEQ_LEN absoluto de 512 — INCORRECTO: Gemini dijo que el límite absoluto de Hannah es 512 tokens (del DPO) y recomendó bajar el contexto RAG a 300 tokens. En realidad, SEQ_LEN=512 del DPO es solo la longitud de entrenamiento de esa fase (por restricción de VRAM al cargar 2 modelos). En inferencia, Hannah procesa hasta 1024 tokens (SEQ_LEN del pretraining y SFT). Nuestro límite de ~200 tokens para Fast es correcto según la arquitectura oficial (Sección 2.5).

2. El RAG NO debe construir el prompt completo: Gemini inyectó [SYS], [USR], [ASS] y el System Prompt dentro del RAG. Según la arquitectura (Sección 2.2), eso es responsabilidad del Token Sequence Handler (Pedro/Marlow). El RAG solo devuelve [MEMORY]...contexto...[/MEMORY].

3. No mencionó [MEMORY]/[/MEMORY]: El tokenizer de Hannah tiene estos tokens especiales específicamente para contexto RAG. Gemini los ignoró completamente.

4. No implementó Semantic Cache: Componente clave de la arquitectura (Sección 2.3) que Gemini omitió.

5. No implementó modos dual (Simplified/Extended): La arquitectura define dos modos de operación con configuraciones distintas. Gemini usó solo una función genérica.

6. No sabe del cambio a Qwen2.5-14B: Sigue mencionando Llama como Slow Model. El Laboratorio Grupal (9 abril 2026) confirma Qwen2.5-14B-Instruct.
